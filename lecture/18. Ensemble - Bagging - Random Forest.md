## 1. Động lực: Tại sao cần Ensemble?

Hãy nghĩ đến một tình huống thực tế: khi cần đưa ra một quyết định quan trọng, chúng ta thường **hỏi ý kiến nhiều người** thay vì chỉ tin vào một chuyên gia duy nhất. Tổng hợp nhiều ý kiến thường cho kết quả tốt hơn bất kỳ cá nhân nào.

**Ensemble Learning** áp dụng triết lý này vào học máy: thay vì xây dựng **một mô hình mạnh**, ta xây dựng **nhiều mô hình yếu (weak learners)** và kết hợp dự đoán của chúng lại.

### Tại sao nhiều mô hình yếu lại tốt hơn một mô hình mạnh?

Giả sử ta có 3 mô hình phân loại, mỗi mô hình có xác suất đúng là **p = 0.7**. Nếu ta dùng **đa số phiếu** (majority vote):

```
P(ít nhất 2 trong 3 đúng) = C(3,2)×0.7²×0.3¹ + C(3,3)×0.7³×0.3⁰
                           = 3×0.49×0.3 + 1×0.343
                           = 0.441 + 0.343 = 0.784
```

Tức là từ 70% → **78.4%** chỉ bằng cách kết hợp! Điều kiện quan trọng: các mô hình phải **đủ khác nhau** (diverse) và **độc lập tương đối** với nhau.

## 3. Bagging — Bootstrap Aggregating

### 3.1. Ý tưởng

**Bagging** sinh ra sự đa dạng giữa các mô hình bằng cách cho mỗi mô hình thấy **một phiên bản hơi khác** của tập dữ liệu.

**Quy trình:**

```
Dữ liệu gốc (N quan sát)
    │
    ├── Bootstrap sample 1 → Mô hình 1 → Dự đoán 1
    ├── Bootstrap sample 2 → Mô hình 2 → Dự đoán 2
    ├── ...
    └── Bootstrap sample B → Mô hình B → Dự đoán B
                                              │
                              Kết hợp: Majority Vote (phân loại)
                                        hoặc Average (hồi quy)
```

**Bootstrap sample** là gì? Lấy mẫu **có hoàn lại (with replacement)** N quan sát từ N quan sát gốc. Kết quả: ~63.2% quan sát gốc xuất hiện ít nhất một lần, ~36.8% không được chọn.

### 3.2. Công thức kết hợp

**Phân loại (Classification):**

$$\hat{y} = \text{mode}\left\{ \hat{y}^{(1)}, \hat{y}^{(2)}, \ldots, \hat{y}^{(B)} \right\}$$

Hoặc dùng xác suất trung bình:

$$\hat{P}(y=1 \mid x) = \frac{1}{B} \sum_{b=1}^{B} \hat{P}^{(b)}(y=1 \mid x)$$

**Hồi quy (Regression):**

$$\hat{y} = \frac{1}{B} \sum_{b=1}^{B} \hat{y}^{(b)}$$

### 3.3. Out-of-Bag (OOB) Error

Một tính năng đặc biệt của Bagging: mỗi bootstrap sample bỏ qua ~36.8% dữ liệu gốc. Những quan sát này (gọi là **Out-of-Bag**) có thể được dùng để **đánh giá mô hình mà không cần tách tập test riêng**!

$$\text{OOB Error} = \frac{1}{N} \sum_{i=1}^{N} \mathbf{1}\left[\hat{y}_i^{\text{OOB}} \neq y_i\right]$$

Trong đó $\hat{y}_i^{\text{OOB}}$ là dự đoán trung bình từ tất cả các cây mà quan sát $i$ **không** nằm trong bootstrap sample.

> 💡 **OOB Error là một dạng CV miễn phí** — kết quả tương đương leave-one-out CV nhưng không tốn thêm thời gian tính toán.

## 4. Random Forest

### 4.1. Vấn đề của Bagging thuần túy

Với Bagging trên Decision Tree, các cây vẫn có thể **tương quan cao** với nhau nếu có một vài biến rất mạnh — mọi cây đều sẽ chọn những biến đó ở các nút đầu tiên.

Khi các cây tương quan → trung bình của chúng không giảm được nhiều variance.

### 4.2. Giải pháp: Random Feature Subsampling

**Random Forest** thêm một bước ngẫu nhiên hóa nữa: tại **mỗi nút phân chia** của mỗi cây, thay vì xét **tất cả** các biến, chỉ xét **một tập con ngẫu nhiên** gồm $m$ biến.

$$m \approx \sqrt{p} \quad \text{(phân loại)} \qquad m \approx \frac{p}{3} \quad \text{(hồi quy)}$$

trong đó $p$ là tổng số biến.

**Kết quả:** Các cây ít tương quan hơn → kết hợp hiệu quả hơn → giảm Variance tốt hơn Bagging thuần túy.

### 4.3. So sánh Bagging vs Random Forest

| | Bagging (Decision Tree) | Random Forest |
|---|---|---|
| Bootstrap sampling | ✅ | ✅ |
| Random feature subset tại mỗi nút | ❌ (dùng tất cả biến) | ✅ ($m < p$) |
| Mức độ tương quan giữa cây | Cao hơn | Thấp hơn |
| Hiệu năng thực tế | Tốt | Tốt hơn |

### 4.4. Siêu tham số quan trọng

| Tham số | Ý nghĩa | Giá trị thường dùng |
|---|---|---|
| `ntree` | Số cây trong rừng | 100–500 (càng nhiều càng ổn định) |
| `mtry` | Số biến xét tại mỗi nút | `sqrt(p)` (phân loại), `p/3` (hồi quy) |
| `nodesize` | Số quan sát tối thiểu tại nút lá | 1 (phân loại), 5 (hồi quy) |
| `maxnodes` | Số nút lá tối đa | NULL (không giới hạn) |

## 5. Feature Importance (Độ quan trọng của biến)

Random Forest cung cấp hai cách đo độ quan trọng của biến:

### Mean Decrease in Impurity (MDI)
Tổng mức giảm Gini Impurity (hoặc MSE) do biến $X_j$ gây ra, trung bình trên tất cả các cây:

$$\text{Importance}(X_j) = \frac{1}{B} \sum_{b=1}^{B} \sum_{t \in \mathcal{T}_b : X_j} \Delta \text{Impurity}(t)$$

**Ưu điểm:** Nhanh, tính trong quá trình huấn luyện.  
**Nhược điểm:** Có thể thiên vị với biến có nhiều giá trị (high cardinality).

### Mean Decrease in Accuracy (MDA / Permutation Importance)
Đo mức giảm accuracy khi **xáo trộn ngẫu nhiên (permute)** giá trị của biến $X_j$ trong OOB data:

$$\text{Importance}(X_j) = \frac{1}{B} \sum_{b=1}^{B} \left[ \text{Acc}^{(b)}_{\text{OOB}} - \text{Acc}^{(b)}_{\text{OOB, permuted } X_j} \right]$$

**Ưu điểm:** Đáng tin cậy hơn MDI.  
**Nhược điểm:** Chậm hơn (cần tính lại sau khi permute).

### 6.1. Bagging với Decision Tree — Cài đặt thủ công

Trước khi dùng package, ta cài đặt Bagging từ đầu để hiểu rõ cơ chế:

```r
library(rpart)  # Decision Tree

titanic <- read.csv("titanic.csv", stringsAsFactors = TRUE)

titanic_clean <- titanic %>%
  select(survived, pclass, sex, age, sibsp, parch, fare) %>%
  mutate(survived = factor(survived, levels = c(0, 1),
                           labels = c("No", "Yes"))) %>%
  drop_na()

set.seed(42)
n     <- nrow(titanic_clean)
B     <- 100  # Số cây

# Lưu trữ dự đoán từ từng cây
bag_preds <- matrix(NA, nrow = n, ncol = B)

for (b in 1:B) {
  # Bước 1: Bootstrap sample
  boot_idx <- sample(1:n, size = n, replace = TRUE)
  boot_data <- titanic_clean[boot_idx, ]

  # Bước 2: Huấn luyện Decision Tree trên bootstrap sample
  tree_b <- rpart(survived ~ ., data = boot_data, method = "class")

  # Bước 3: Dự đoán trên toàn bộ dữ liệu
  bag_preds[, b] <- predict(tree_b, newdata = titanic_clean,
                             type = "class") == "Yes"
}

# Bước 4: Kết hợp bằng majority vote
majority_vote <- rowMeans(bag_preds) >= 0.5
bag_final     <- factor(ifelse(majority_vote, "Yes", "No"))

# Đánh giá
acc_single_tree <- mean(
  predict(rpart(survived ~ ., data = titanic_clean, method = "class"),
          type = "class") == titanic_clean$survived
)
acc_bagging <- mean(bag_final == titanic_clean$survived)

cat(sprintf("Accuracy — Single Tree: %.4f\n", acc_single_tree))
cat(sprintf("Accuracy — Bagging (%d trees): %.4f\n", B, acc_bagging))
```

> 💡 **Quan sát:** Bagging thường cho accuracy cao hơn Single Tree, đặc biệt khi cây đơn lẻ có variance cao (deep tree).

### 6.2. Random Forest với `randomForest`

#### Dữ liệu: German Credit

```r
german <- read.csv("german1.csv")
german$target <- factor(german$target, levels = c(1, 2),

train_g   <- german[train_idx, ]
test_g    <- german[-train_idx, ]

cat("Tập train:", nrow(train_g), "| Tập test:", nrow(test_g), "\n")
```

#### Huấn luyện Random Forest cơ bản

```r
library(randomForest)

set.seed(42)
rf_model <- randomForest(
  target   ~ .,
  data     = train_g,
  ntree    = 500,
  mtry     = floor(sqrt(ncol(train_g) - 1)),  # sqrt(p) cho phân loại
  importance = TRUE,   # Tính feature importance
  proximity  = TRUE    # Tính ma trận proximity (dùng sau)
)

print(rf_model)
```

**Đọc output:**

```

               Type of random forest: classification
                     Number of trees: 500
No. of variables tried at each split: 5

        OOB estimate of error rate: ~25%
Confusion matrix:
     Good Bad class.error
Good  xxx  xx       ~0.10
Bad    xx  xx       ~0.40
```

> 📌 **OOB Error** là ước lượng tự động, không cần tách tập validation riêng.

#### Đánh giá trên tập test

```r
# Dự đoán nhãn và xác suất
pred_class <- predict(rf_model, newdata = test_g, type = "class")

# Accuracy & Confusion Matrix
cm <- confusionMatrix(pred_class, test_g$target, positive = "Bad")

# AUC
roc_rf <- roc(test_g$target, pred_prob,
              levels = c("Good", "Bad"), quiet = TRUE)
cat("AUC (Random Forest):", round(auc(roc_rf), 4), "\n")
```

```r
library(tidymodels)

# Bước 1: Chuẩn bị dữ liệu
german_split <- initial_split(german, prop = 0.8, strata = target)
german_train <- training(german_split)
german_test  <- testing(german_split)

# Cross-validation folds
german_folds <- vfold_cv(german_train, v = 5, strata = target)

# Bước 2: Định nghĩa recipe (tiền xử lý)
rf_recipe <- recipe(target ~ ., data = german_train) %>%
  step_unknown(all_nominal_predictors()) %>%
  step_dummy(all_nominal_predictors())

# Bước 3: Định nghĩa model (với tham số cần tune)
rf_spec <- rand_forest(
  mtry  = tune(),    # Sẽ tìm giá trị tối ưu
  trees = 300,
  min_n = tune()
) %>%
  set_engine("ranger", importance = "impurity") %>%
  set_mode("classification")

# Bước 4: Kết hợp thành workflow
rf_workflow <- workflow() %>%
  add_recipe(rf_recipe) %>%
  add_model(rf_spec)

# Bước 5: Định nghĩa lưới tham số
rf_grid <- grid_regular(
  mtry(range = c(3, 15)),
  min_n(range = c(2, 10)),
  levels = 4
)

# Bước 6: Tune
set.seed(42)
rf_tune_results <- tune_grid(
  rf_workflow,
  resamples = german_folds,
  grid      = rf_grid,
  metrics   = metric_set(roc_auc, accuracy)
)

# Xem kết quả
collect_metrics(rf_tune_results) %>%
  filter(.metric == "roc_auc") %>%
  arrange(desc(mean)) %>%
  head(10)

# Vẽ đồ thị
autoplot(rf_tune_results)

# Chọn tham số tốt nhất
best_params <- select_best(rf_tune_results, metric = "roc_auc")
cat("Tham số tốt nhất:\n")
print(best_params)
```

```r
medical <- read.csv("medical_care.csv", stringsAsFactors = TRUE)

  data       = med_train,

               levels = c("No", "Yes"), quiet = TRUE)

                FEMER + FDOCT + UIMMSTAT + UAGE + U_FTPT + U_WKSLY +
                UBRACE + GENDER + UEDUC3,
              data = med_train, family = binomial())
pred_lr <- predict(lr_med, newdata = med_test, type = "response")
roc_lr  <- roc(med_test$UCURNINS, pred_lr,
               levels = c("No", "Yes"), quiet = TRUE)

cat("AUC (Logistic Regression):", round(auc(roc_lr), 4), "\n")
cat("Cải thiện:", round(auc(roc_med) - auc(roc_lr), 4), "\n")
```

### 6.7. Wines — So sánh toàn diện các mô hình

```r
wines <- read.csv("wines.csv", stringsAsFactors = TRUE)
wines$type <- factor(wines$type)

# Loại biến quality để tránh data leakage (quality liên quan type)
wines_feat <- wines %>% select(-quality)

wines_split <- initial_split(wines_feat, prop = 0.8, strata = type)
wines_train <- training(wines_split)
wines_test  <- testing(wines_split)

# Hàm tính AUC tiện lợi
get_auc <- function(model, test_data, truth_col, prob_col) {
  preds <- predict(model, newdata = test_data, type = "prob")[, prob_col]
  roc_obj <- roc(test_data[[truth_col]], preds,
                 levels = levels(test_data[[truth_col]]), quiet = TRUE)
  round(auc(roc_obj), 4)
}

ctrl_compare <- trainControl(
  method = "cv", number = 5,
  classProbs = TRUE, summaryFunction = twoClassSummary
)

set.seed(42)

# Logistic Regression (baseline)
m_lr <- train(type ~ ., data = wines_train, method = "glm",
              family = "binomial", trControl = ctrl_compare, metric = "ROC")

# Single Decision Tree
m_tree <- train(type ~ ., data = wines_train, method = "rpart",
                trControl = ctrl_compare, metric = "ROC")

# Bagging
m_bag <- train(type ~ ., data = wines_train, method = "treebag",
               trControl = ctrl_compare, metric = "ROC")

# Random Forest
m_rf <- train(type ~ ., data = wines_train, method = "rf",
              trControl = ctrl_compare, metric = "ROC",
              tuneGrid = data.frame(mtry = c(3, 4, 5)))

# Tổng hợp kết quả CV
results_cv <- resamples(list(
  LogisticReg   = m_lr,
  DecisionTree  = m_tree,
  Bagging       = m_bag,
  RandomForest  = m_rf
))
```

### 6.9. Titanic — Quy trình hoàn chỉnh với `tidymodels`

```r
titanic <- read.csv("titanic.csv", stringsAsFactors = TRUE)

titanic_clean <- titanic %>%
  select(survived, pclass, sex, age, sibsp, parch, fare) %>%
  mutate(survived = factor(survived, levels = c(0, 1),
                           labels = c("No", "Yes"))) %>%
  drop_na()

# Split
set.seed(42)
tit_split <- initial_split(titanic_clean, prop = 0.8, strata = survived)
tit_train <- training(tit_split)
tit_test  <- testing(tit_split)
tit_folds <- vfold_cv(tit_train, v = 5, strata = survived)

# Recipe
tit_recipe <- recipe(survived ~ ., data = tit_train) %>%
  step_dummy(all_nominal_predictors()) %>%
  step_normalize(all_numeric_predictors())

# Random Forest spec
rf_tit_spec <- rand_forest(
  mtry  = tune(),
  trees = 300,
  min_n = tune()
) %>%
  set_engine("ranger", importance = "permutation") %>%
  set_mode("classification")

# Workflow
rf_tit_wf <- workflow() %>%
  add_recipe(tit_recipe) %>%
  add_model(rf_tit_spec)

# Tune
set.seed(42)
tit_tune <- tune_grid(
  rf_tit_wf,
  resamples = tit_folds,
  grid      = grid_regular(mtry(range = c(1, 5)),
                            min_n(range = c(2, 20)), levels = 4),
  metrics   = metric_set(roc_auc)
)

# Best params
best_tit <- select_best(tit_tune, metric = "roc_auc")
cat("Tham số tốt nhất cho Titanic RF:\n")
print(best_tit)

# Final evaluation
final_tit <- last_fit(finalize_workflow(rf_tit_wf, best_tit), tit_split)
collect_metrics(final_tit)

# Feature importance
final_tit %>%
  extract_fit_parsnip() %>%
  vip(num_features = 7,
      aesthetics = list(fill = "steelblue")) +

  theme_minimal()
```

## 7. So sánh `randomForest` vs `ranger` vs `caret` vs `tidymodels`

| | `randomForest` | `ranger` | `caret` | `tidymodels` |
|---|---|---|---|---|
| Tốc độ | Chậm | **Rất nhanh** | Phụ thuộc engine | Phụ thuộc engine |
| Parallel | Giới hạn | ✅ Tốt | ✅ | ✅ |
| Proximity matrix | ✅ | Giới hạn | ❌ | ❌ |
| Tuning | Thủ công | Thủ công | ✅ Tích hợp | ✅ Tích hợp |
| Tidyverse friendly | ❌ | Một phần | Một phần | ✅ |
| Phù hợp | Học cơ chế, proximity | Dữ liệu lớn | Nhất quán Bài 17 | Workflow hoàn chỉnh |

> 💡 **Khuyến nghị thực tế:**
> - **Học hiểu cơ chế** → `randomForest`
> - **Dữ liệu lớn, production** → `ranger` qua `tidymodels`
> - **Nhất quán với bài trước** → `caret`

## 8. Tổng kết — Khi nào dùng Random Forest?

**Random Forest phù hợp khi:**
- Dữ liệu có nhiều biến, không biết biến nào quan trọng (feature importance giúp lọc).
- Có cả biến số (numeric) và biến phân loại (categorical).
- Không muốn tiền xử lý phức tạp (RF không cần chuẩn hóa, ít nhạy outlier).
- Cần đánh giá nhanh không cần tách validation set (dùng OOB error).

**Random Forest ít phù hợp khi:**
- Dữ liệu rất thưa thớt (sparse) như văn bản, ma trận người dùng–sản phẩm.
- Cần mô hình có thể giải thích từng bước (explainability cao) → dùng Logistic Regression hoặc Decision Tree đơn.
- Tài nguyên tính toán rất hạn chế (mô hình lớn, chậm khi predict).
